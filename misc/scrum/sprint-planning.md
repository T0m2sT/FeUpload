# Sprint Planning and Sprint Backlog Rules

This document defines how the team applies the patterns requested in class.

## Patterns used
- Definition of Ready (DoR)
- Sprint Planning
- Sprint Backlog
- Small Items
- Yesterday's Weather
- Pigs Estimate

## Definition of Ready (DoR)
An item can be pulled from Product Backlog to Sprint Backlog only when:
1. User value is clear.
2. Acceptance criteria exist.
3. Dependencies are identified.
4. Estimate exists (Pigs Estimate).
5. Item is small enough to complete within one sprint.

## Sprint Planning workflow
1. Start from Product Backlog ordered by priority.
2. Select only ready items (DoR met).
3. Size items with Pigs Estimate.
4. Use Yesterday's Weather to set realistic sprint capacity.
5. Enforce Small Items rule (split large stories).
6. Build Sprint Backlog with no epics.

## Sprint Backlog execution rules
- Items are moved from Product Backlog during Sprint Planning.
- Assignee(s) are set only when item enters `In Progress`.
- Work starts in pair/trio/team, not by pre-allocation.
- WiP is kept low; avoid paused work.

## Sprint Backlog quality checks
- No epic-sized items.
- Scope fits prior sprint velocity.
- Each item has acceptance criteria and test intent.
- Item has owner(s) only after start.
